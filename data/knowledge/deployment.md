# 大模型部署与推理优化

## 推理的两个阶段

### Prefill（预填充）
- 输入完整的prompt，一次性计算所有token的注意力
- 计算量大，吞吐量敏感
- 时间与输入长度平方成正比 O(n²)

### Decode（解码）
- 逐token生成，每次只算新token对历史的注意力
- 延迟敏感
- 依赖KV Cache加速

## KV Cache
- 在Decode阶段，每个新token只需要计算对之前所有token的注意力
- 把每层的K和V矩阵缓存起来，后续token直接读取
- 节省的计算量：O(n²) → O(n)
- 代价：显存占用增加

## 量化技术

### INT8 / INT4 量化
- 将FP16权重压缩到8-bit或4-bit
- INT4：显存需求降至原来的1/4
- 分组量化：每64个权重一组独立缩放，减少精度损失

### GPTQ / AWQ
- GPTQ：基于OBQ的逐层量化，需要校准数据
- AWQ：通过分析激活值分布，保护重要通道

### GGUF (llama.cpp格式)
- 专为CPU推理优化的量化格式
- 支持在普通电脑上运行7B甚至13B模型
- Ollama底层使用的就是GGUF

## GQA (分组查询注意力)
- Q头数多（如32），K和V头数少（如8）
- 多个Q头共享同一组KV
- 显著减少KV Cache的显存占用
- Qwen2.5-3B: Q头=32, KV头=8

## Flash Attention
- 不把完整的注意力矩阵存到显存
- 分块计算softmax，边算边累积
- O(n²)显存 → O(n)

## 显存全景（以Qwen2.5-3B为例）
| 占用项 | 大小 |
|---|---|
| 模型权重(FP16) | ~6GB |
| KV Cache(1024 tokens) | ~0.5GB |
| 激活值(推理时) | <0.5GB |
| 总计 | ~7GB |
